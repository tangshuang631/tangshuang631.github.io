# 个人展示页指南

## 文档定位

这是一份长期使用的个人展示页统一规范

后续首页结构、项目展示页、文案语气、截图排布、动效和信息层级都以本文件为准

如果后面继续加入新项目，不再单独新建一套临时风格说明，优先沿用这里的规则

## 总体目标

个人主页要做成正式产品官网风格，不做传统作品集堆砌，也不做技术博客首页

页面需要同时满足三件事

- 能快速建立个人身份感
- 能突出一个当前最有代表性的重点项目
- 能让后续其他项目以同一套方式继续加入，整体风格不散

## 站点结构

整个站点采用两层结构

1. 首页负责总览
2. 独立项目页负责展开

结构原则

- 首页保持收敛
- 重点项目集中展示
- 详细信息放到独立页面
- 一个项目一页，避免把首页做成信息仓库

## 首页规范

首页不是项目目录墙，而是一个克制的产品 Landing 页

### 首页职责

- 展示姓名与基础定位
- 给出当前最值得看的项目入口
- 提供外部链接与联系入口
- 控制信息量，让用户第一屏就知道该点哪里

### 首页信息结构

1. 顶部导航
2. 简短首屏介绍
3. 重点项目入口
4. 页脚入口

### 首页内容原则

- 只保留少量核心信息
- `OpenCow` 作为当前第一重点项目放在最前
- 其他项目弱化处理，只保留必要入口
- 不堆叠多个项目卡片
- 不在首页展开技术细节
- 不在首页写成长篇项目说明

### 首页视觉原则

- 像正式产品首页，不像学生作品集模板
- 使用白色和浅灰作为基础色
- 用细边框和灰色分割线组织层次
- 保留留白，但不能空
- 圆角克制，避免强卡片感
- 动画少量即可，只保留必要入场节奏

## 独立项目页规范

每个重点项目都使用统一的长页面结构，通过横向模块和分割线建立秩序

### 标准结构

1. Hero
2. 产品概览
3. 模块展示
4. 当前状态
5. 返回入口

### Hero 规范

- 用一句话说明产品定位
- 用一小段文字交代产品范围
- 只保留必要动作按钮，例如 `GitHub`、`查看模块`、`返回主页`
- 辅助文案只说明当前阶段，不写空泛宣传语

### 产品概览规范

- 只回答产品是什么
- 只回答为什么这样组织
- 用短句说明核心目标
- 不写冗长背景故事

### 模块展示规范

模块展示是独立项目页的核心部分

每个模块必须同时包含

- 真实截图
- 模块名称
- 技术栈
- 功能说明
- 设计思路或后续方向

每个截图旁边都必须能回答三个问题

1. 这个模块在做什么
2. 这个模块由什么能力支撑
3. 这个模块为什么这样设计

### 当前状态规范

- 用正式语言说明当前完成度
- 可以直接写前端和后端仍在优化
- 可以直接写下一阶段重点
- 不写悬浮表达

## OpenCow 作为基准案例

`OpenCow` 是当前整站的基准展示项目

以后新增项目时，默认参考 `OpenCow` 的表达方式、模块组织方式和信息密度

### OpenCow 首页入口口径

- 面向本地工作流的桌面工作台产品
- 统一承载对话、搜索、知识管理与扩展能力接入

### OpenCow 独立页标准模块

- 会话工作区
- 搜索与 RAG
- 知识库
- Skills
- NPC
- MCP
- 审计
- 设置

### OpenCow 各模块表达重点

会话工作区

- 技术栈要强调 React、Tauri、Ollama、本地模型交互
- 功能要说明输入、响应、上下文和结果输出承载关系
- 设计要说明左侧导航、中间工作区、右侧状态区的组织方式

搜索与 RAG

- 技术栈要强调搜索来源配置、自定义 Provider、RAG 上下文接入
- 功能要说明搜索配置、测试、来源展示与回退能力
- 设计要说明为什么把搜索做成显式可配置入口，而不是隐藏能力

知识库

- 技术栈要强调本地文件导入、共享文件库、知识库绑定
- 功能要说明文件整理、知识绑定与后续角色复用
- 设计要说明先整理共享文件再绑定知识库的组织逻辑

Skills

- 技术栈要强调本地技能目录、安装、启用与管理
- 功能要说明推荐技能、已安装技能和后续操作入口
- 设计要说明为什么把技能单独管理，而不是藏在聊天流程里

NPC

- 技术栈要强调角色配置、知识库绑定、本地模型协作
- 功能要说明不同角色如何调用不同知识和能力
- 设计要说明角色定位、知识来源和能力边界拆分

MCP

- 技术栈要强调 MCP 管理、推荐清单、扩展接入
- 功能要说明可安装 MCP 和扩展入口
- 后续方向要明确写浏览器联动、网页读取和更多工具型扩展

审计

- 技术栈要强调执行日志、审计记录、任务轨迹展示
- 功能要说明关键行为和上下文来源记录
- 设计要说明前台可查看日志对排查和回溯的价值

设置

- 技术栈要强调 Ollama 本地模型配置和全局参数设置
- 功能要说明模型检测、输出预算、模型切换和全局配置入口
- 设计要说明系统级能力统一收口的必要性

## 视觉系统规范

### 色彩

- 以白色和浅灰为主
- 不依赖强烈渐变制造设计感
- 重点依靠层级、边框、留白和排版建立质感

### 布局

- 多使用完整段落和横向模块
- 多使用灰色分割线
- 少用卡片
- 避免卡片套卡片
- 避免页面被分割得太碎

### 排版

- 标题使用衬线字体
- 正文使用清晰的无衬线字体
- 字号层级明确
- 段落短
- 信息密度高但不拥挤

### 动效

- 只保留轻量 reveal
- 不做大面积视差
- 不做无意义漂浮和装饰动画
- 任何动画都不能影响首屏可见性和加载稳定性

### 性能边界

- 页面不能因为动画依赖缺失而变空白
- 页面不能因为无意义动画过多而卡顿
- 图片展示优先真实内容，不用复杂包裹结构增加负担

## 文案规范

### 语气

- 正式
- 清楚
- 像成熟产品页面
- 不像项目汇报
- 不像 AI 自动生成的营销文案

### 文案规则

- 中文为主，保留必要英文产品名和技术术语
- 句子短
- 直接说产品和能力
- 不写口号式空话
- 不写明显夸张表达
- 页面文案不使用中文句号

### 推荐表达

- 面向本地工作流的桌面工作台产品
- 统一承载对话、搜索、知识管理与扩展能力接入
- 支持文件导入、知识库绑定与后续角色复用
- 继续完善前端细节、后端链路稳定性与扩展能力接入

### 明确避免

- 持续优化中
- 已可用
- 核心亮点
- 完整闭环
- 高效赋能
- 这个页面很酷
- 这个模块很强

## 新项目接入规则

以后新增其他项目时，先判断它是否值得进入首页重点位置

如果是重点项目，按下面流程处理

1. 首页增加一个简洁入口
2. 新建独立项目页
3. 沿用本指南的统一结构
4. 为该项目补充真实截图
5. 每张截图旁边写技术栈、功能、设计思路
6. 用正式文案收口当前状态和后续方向

如果不是重点项目

- 只保留外链或次级入口
- 不破坏首页的收敛感

## 更新检查清单

每次更新首页或项目页后，都要检查以下内容

1. 页面有没有重新出现大面积卡片化
2. 首页有没有被过多说明挤满
3. 项目页每个截图旁边是否有技术栈、功能和设计说明
4. 文案是否变得口语化、奇怪或 AI 味过重
5. 是否重新出现中文句号
6. 动画是否影响首屏显示
7. 页面是否因为依赖问题变空白
8. 新项目是否仍然和 `OpenCow` 保持同一套展示逻辑
