# 个人主页项目展示风格规范

## 适用范围

适用于首页重点项目入口、独立项目展示页，以及后续新增的案例型页面

## 总体目标

个人主页不是传统项目列表页，也不是技术博客首页

目标是让页面更接近正式产品官网

- 首页负责建立身份感与重点项目入口
- 项目页负责完整展示产品结构、关键功能和实现方向
- 页面既要面向用户，也要让招聘方或合作方快速理解项目完成度

## 首页规范

- 首页只保留少量核心信息
- `OpenCow` 作为当前第一重点项目放在最前
- 不做多项目并列卡片墙
- 不在首页堆叠大量项目说明
- `codex-loop` 等其他项目弱化为次级入口或外链

首页文案要求

- 用词正式
- 句子短
- 不写口号式空话
- 不写过度压缩的奇怪短句
- 不使用明显 AI 味标题

首页结构要求

1. 顶部导航
2. 简短首屏介绍
3. 单个重点项目入口
4. 联系方式与其他仓库入口

## 项目展示页规范

项目展示页采用长页面结构，通过灰色分割线切出段落，不依赖卡片堆叠做层次

标准结构

1. Hero
2. 产品概览
3. 模块展示
4. 当前状态
5. 返回首页与仓库入口

### Hero

- 一句话说明产品定位
- 用正式表达说明当前产品范围
- 按钮只保留必要动作，例如 `GitHub` 和 `查看模块`
- 辅助说明用于交代当前阶段，不写夸张宣传语

### 产品概览

- 只回答产品是什么
- 只回答为什么这样组织
- 不写大段背景故事

### 模块展示

每个模块都使用同一套表达框架

- 真实截图
- 模块名称
- 技术栈
- 功能说明
- 设计思路或后续方向

每个截图旁边必须能回答三个问题

1. 这个模块在做什么
2. 这个模块由什么能力支撑
3. 这个模块为什么这样设计

建议模块口径

- 会话工作区：强调本地模型交互、主工作区承载关系
- 搜索与 RAG：强调搜索来源配置、上下文接入、失败回退与设计思路
- 知识库：强调文件导入、共享文件与复用关系
- Skills：强调能力管理，而不是把能力藏在聊天流程里
- NPC：强调角色、知识与能力边界拆分
- MCP：强调可扩展工具接入，以及后续浏览器联动、网页读取、更多工具型扩展方向
- 审计：强调执行记录、回溯与排查价值
- 设置：强调系统级配置统一收口

### 当前状态

- 用正式方式说明产品已完成到什么程度
- 可以直接写前端和后端仍在优化
- 不使用“可用”“持续优化中”这类口语化或悬浮表达

## 视觉规范

- 以白色和浅灰为主
- 通过细边框和分割线建立秩序
- 少量动画即可，动画只负责入场和节奏
- 不做大面积悬浮动效
- 不做复杂视差和无意义装饰
- 圆角尽量克制，避免强卡片感

明确避免

- 卡片套卡片
- 过多信息块并排
- 夸张渐变
- 重阴影
- 为了设计感而增加无信息价值组件

## 文案规范

- 中文为主，保留必要英文产品名和技术术语
- 语气正式，接近成熟产品页面
- 不写过度营销化表达
- 不写项目汇报腔
- 不写“这个页面很酷”“这个模块很强”这类主观句
- 页面文案不使用中文句号

推荐表达方式

- 面向本地工作流的桌面工作台产品
- 统一承载对话、搜索、知识管理与扩展能力接入
- 支持文件导入、知识库绑定与后续角色复用
- 继续完善前端细节、后端链路稳定性与扩展能力接入

不推荐表达方式

- 持续优化中
- 已可用
- 核心亮点
- 完整闭环
- 高效赋能

## 后续执行规则

- 新增项目展示页时，先参考本规范，再参考 `OpenCow` 当前页面结构
- 如果需要新增视觉元素，优先先问这个元素是否真的在传递信息
- 如果一个区块删掉以后不影响理解，优先删掉
- 页面更新以后，先检查是否变空白、是否因为动画依赖导致内容隐藏、是否重新出现强卡片化倾向
