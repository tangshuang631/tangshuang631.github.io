# OpenCow 项目展示设计说明

## 目标

在现有个人主页中加入一个更完整的代表性项目展示入口，让 `opencow` 成为首页中的重点项目，并通过独立案例页承载更深入的设计展示、关键功能说明和工程实现信息。

## 设计方向

- 展示模式：首页重点入口 + 独立案例页
- 内容比例：设计展示与功能说明各占一半
- 页面气质：延续当前主页的克制、干净和正式风格，同时让 `opencow` 更接近真实产品展示页
- 内容语言：中文为主，保留必要的英文项目名与技术术语
- 详细风格：以后统一遵循 `2026-06-22-portfolio-showcase-style-guide.md`

## 信息结构

### 首页

首页继续保持轻量结构，只保留一个更强的 `opencow` 项目入口，不把完整案例内容直接塞进首页。

首页入口需要包含：

- 项目分类：`Web / 全栈开发`
- 项目名称：`opencow`
- 一句定位：突出它是一个面向本地工作流与 AI 协作的桌面工作台/智能工作台
- 亮点标签：如 `Workbench`、`RAG/Search`、`Desktop UX`、`Local-first`
- 操作按钮：`查看案例页` 与 `GitHub`

### 独立案例页

案例页使用单页滚动结构，包含以下模块：

1. Hero
   - 项目名、项目定位、返回首页、仓库入口
   - 一段短说明，概括 `opencow` 的设计目标和产品气质
2. 产品概览
   - 用高密度但不拥挤的概览区说明这是什么项目、解决什么问题、主要使用场景
3. 模块展示
   - 使用真实截图
   - 每个模块都补充技术栈、功能说明与设计思路
   - 重点展示左侧导航、中间工作区、右侧状态区的组织方式
4. 关键功能
   - 会话工作流
   - 搜索 / RAG 能力
   - 会话持久化
   - 权限确认与回退机制
   - Skills / NPC / MCP 等本地能力管理
5. 工程实现
   - 说明该项目的技术栈、桌面端集成方式、状态管理思路与本地优先策略
6. 总结与入口
   - 总结项目代表的能力
   - 保留仓库访问与返回首页入口

## 内容选择原则

- 不虚构未确认的线上成果、发布数据或用户规模
- 以当前仓库中已能确认的前端结构和功能方向为基础提炼表达
- 优先展示 `Workbench`、搜索/RAG、会话恢复、本地模型与受控能力这些能体现产品完成度的部分
- 避免写成纯技术文档，也避免只做泛泛而谈的作品集包装

## 视觉要求

- 延续当前主页的柔和浅色体系，但 `opencow` 案例页应更偏产品展示页，而不是普通简历页
- 减少重复卡片堆叠，更多使用长分区、分割线、信息栏和横向模块排布
- 标题保持衬线气质，正文保持清晰可读的无衬线排版
- 通过边框、分割线和留白体现产品感，避免依赖夸张动效和重装饰
- 模块密度要高于主页首页，同时在移动端仍然保持清楚

## 实现策略

- 首页修改：调整项目分类和项目目录中的 `Web / 全栈开发` 区域，让 `opencow` 成为正式项目入口
- 新增页面：创建 `opencow.html`
- 样式策略：在现有 `styles.css` 基础上扩展一组可复用的 case study 样式，避免额外引入复杂依赖
- 动效策略：只保留轻量 reveal，避免影响加载与阅读节奏

## 成功标准

- 首页出现清晰的 `opencow` 项目入口，并能跳转到独立案例页
- 案例页能同时传达设计感和功能完成度，而不是只剩下技术名词堆叠
- 页面整体风格与现有个人站点一致，但案例页层次明显更完整
- `opencow` 项目展示足够适合作为简历或作品集中的重点项目入口
